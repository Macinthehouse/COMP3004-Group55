HintonMarket System Workflow Analysis

## SYSTEM OVERVIEW

User interacts with Qt UI → UI calls Controller → Controller updates Domain objects → System returns result → UI updates screen.

User Action
↓
Qt UI (Boundary Class)
↓
Controller (Application Logic)
↓
Domain Objects (Vendor, MarketDay, Booking, Waitlist)
↓
Result returned to UI
↓
UI updates screen / shows message

## SYSTEM SCREENS (UI BOUNDARY CLASSES)

1. Startup Screen

2. Vendor Dashboard

3. Booking Screen

4. Waitlist Status Screen

5. Notifications Panel

6. STARTUP SCREEN

---

UI Class: StartupUI

User sees buttons:

* Register Vendor
* Login Vendor
* Exit

WORKFLOW: REGISTER VENDOR

User clicks:
Register Vendor

System opens:
Vendor Registration Form

Fields:

* Business Name
* Owner Name
* Email
* Phone
* Address
* Vendor Category (Food / Artisan)

User clicks:
Submit

System Action:
VendorController.registerVendor()

System Response:

* Vendor created
* Vendor stored in storage manager

UI Response:

Success popup:
"Vendor registered successfully."

Then redirect to:

Vendor Dashboard

WORKFLOW: LOGIN VENDOR

User enters:

Vendor ID

Clicks:

Login

System Action:

AuthenticationController.loginVendor(vendorID)

System Response:

* Check vendor exists
* Load vendor data

UI Response:

Open Vendor Dashboard

2. VENDOR DASHBOARD

---

UI Class: VendorDashboardUI

User sees:

Vendor Info
Current Bookings
Notifications

Buttons:

* Book Stall
* Cancel Booking
* View Waitlist
* Refresh      ((((((we said this will automatically update)

DASHBOARD → BOOK STALL

User clicks:

Book Stall

Booking Screen

3. BOOKING SCREEN

---

UI Class: BookingUI

User sees:

List of upcoming Market Dates

Example:

* June 1
* June 8
* June 15
* June 22

For each date show:

* Food stalls available
* Artisan stalls available

Buttons:

* Book
* Join Waitlist
* Back

WORKFLOW: BOOK STALL

User selects:

Market Date

Clicks:

Book

UI Action:

BookingController.bookStall(vendorID, marketDateID)

CONTROLLER LOGIC

1. Retrieve Vendor
2. Retrieve MarketDate
3. Check duplicate booking
4. Check stall availability

CASE 1: BOOKING SUCCESSFUL

Controller result:

SUCCESS

System actions:

* Create Booking
* Update Vendor bookings
* Update MarketDate capacity
* Create Notification

UI Response:

Popup:

"Booking successful."

Dashboard refreshes.

Booking appears in booking list.

CASE 2: MARKET FULL

Controller result:

MARKET_FULL

UI Response:

Popup:

"Market is full for this category."

Offer button:

Join Waitlist

WORKFLOW: JOIN WAITLIST

User clicks:

Join Waitlist

UI Action:

WaitlistController.joinWaitlist(vendorID, marketDateID)

System Action:

Add vendor to waitlist queue

UI Response:

Popup:

"You have been added to the waitlist."

DASHBOARD → CANCEL BOOKING (((((follow the image i have sent in the group for the cancel booking)))

User clicks:

Cancel Booking

UI shows:

List of vendor bookings

User selects booking.

Clicks:

Cancel

UI Action:

BookingController.cancelBooking(vendorID, marketDateID)

SYSTEM ACTIONS

Controller:

* Remove booking
* Update market capacity
* Create cancellation notification
* Trigger waitlist promotion

WAITLIST PROMOTION

System checks:

WaitlistController.handlePromotionIfNeeded()

If someone is waiting:

* Next vendor promoted
* Booking automatically created
* Notification sent

UI RESPONSE

Vendor sees popup:

"Booking cancelled successfully."

Dashboard updates.

DASHBOARD → VIEW WAITLIST

User clicks:

View Waitlist

UI shows:

* Waitlist position
* Market date
* Vendor category

Example:

June 8
Position #2

## NOTIFICATIONS PANEL

Dashboard always displays notifications.

Examples:

* Booking confirmed
* Booking cancelled
* Promoted from waitlist

Notification workflow:

Controller creates Notification object.

Vendor stores it:

vendor.addNotification()

UI displays notification list.

## LOGOUT WORKFLOW

User clicks:

Logout

UI Action:

* Close dashboard
* Return to Startup screen

## SYSTEM ERROR HANDLING

UI should handle these errors:

* Vendor not found
* Market date invalid
* Already booked
* Market full
* Waitlist errors

Each error should show a popup using:
